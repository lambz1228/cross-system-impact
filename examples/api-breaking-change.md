# 案例二：REST API 破坏性变更

## 场景描述

订单服务需要重构订单详情接口的响应格式，将嵌套的 `items` 数组扁平化，并将 `total` 字段改名为 `amount`。

**变更内容：**
- 接口：`GET /api/orders/:id`
- 变更类型：破坏性变更（Breaking Change）
- 服务：订单服务
- 文件：`src/controllers/order.controller.ts`

**变更前：**
```json
{
  "id": "ord_123",
  "total": 99.00,
  "status": "paid",
  "items": [
    { "product_id": "p_1", "name": "商品A", "price": 50.00, "quantity": 2 }
  ]
}
```

**变更后：**
```json
{
  "id": "ord_123",
  "amount": 99.00,
  "status": "paid",
  "product_ids": ["p_1"],
  "product_names": ["商品A"],
  "product_prices": [50.00],
  "product_quantities": [2]
}
```

---

## 第一步：变更识别

```
变更对象：
- 接口：GET /api/orders/:id
- 变更类型：破坏性变更
- 变更内容：
  1. total -> amount（字段重命名）
  2. items[] 扁平化为 product_ids[], product_names[], etc.
- 所属服务：订单服务
- 变更文件：src/controllers/order.controller.ts
```

## 第四步：影响评估

| # | 调用方 | 影响等级 | 影响类型 | 失败模式 | 恢复难度 | 检测难度 |
|---|-------|---------|---------|---------|---------|---------|
| 1 | 前端 Web | Critical | 功能中断 | TypeScript 类型错误 + 运行时 undefined | 简单 | 立即 |
| 2 | 移动端 App | Critical | 功能中断 | 类型错误 + 运行时 undefined | 中等 | 延迟 |
| 3 | 小程序 | High | 动态取值 undefined | 显示空白或 0 | 简单 | 延迟 |
| 4 | 客服系统 | High | 数据解析失败 | JSON 解析异常 | 简单 | 立即 |
| 5 | 第三方 ERP | Critical | 集成中断 | Webhook 处理失败 | 复杂 | 延迟 |

**影响统计：**
- Critical: 3 个调用方
- High: 2 个调用方
- **总计影响: 5 个调用方**

---

## 第五步：风险分析

### 前端 Web（Critical）

**依赖方式：** HTTP API 调用，TypeScript 类型定义

**影响分析：**
- TypeScript 编译阶段会报类型错误（好事）
- 如果跳过类型检查，运行时 total 和 items 为 undefined
- 订单详情页完全无法显示

**最坏情况：** 订单详情页白屏

**能否自动检测：** 能，TypeScript 编译会失败

### 移动端 App（Critical）

**依赖方式：** HTTP API 调用，React Native

**影响分析：**
- 与前端类似，但移动端有发布延迟
- 旧版本 App 会使用旧的 API 响应格式
- 新版本 App 需要等待审核发布

**最坏情况：** 旧版本 App 订单详情页崩溃

**能否自动检测：** 能，但修复需要发版

### 第三方 ERP（Critical）

**依赖方式：** Webhook 推送，透传 API 响应

**影响分析：**
- Webhook 处理逻辑依赖 total 和 items 字段
- 字段变更后，ERP 无法正确解析订单数据
- 可能导致库存、财务数据错误

**最坏情况：** 第三方 ERP 数据错误，需要人工修复

**能否自动检测：** 难，需要第三方反馈

## 第六步：行动建议

### [紧急 - 上线前必须完成]

1. **评估是否真的需要破坏性变更**
   - 考虑使用 API 版本管理（v1/v2）
   - 或者同时返回新旧字段，逐步迁移

2. **如果确定要变更，通知所有调用方**
   - 前端组：@前端群
   - 移动组：@移动群
   - 客服组：@客服群
   - 第三方 ERP：@商务联系人

3. **协调发布顺序**
   - 先发布订单服务（保持旧格式）
   - 调用方完成修改后，再切换新格式

### [推荐方案：API 版本管理]

```
方案 A：新增 v2 接口
- 保留 GET /api/v1/orders/:id（旧格式）
- 新增 GET /api/v2/orders/:id（新格式）
- 调用方逐步迁移到 v2
- 6 个月后下线 v1

方案 B：渐进式迁移
- 同时返回新旧字段
- 调用方修改完成后，移除旧字段
```

### [如果必须立即变更]

4. **前端/小程序**：修改 TypeScript 类型和字段引用
5. **移动端**：修改代码 + 发布新版本 + 等待审核
6. **客服系统**：修改 JSON 解析逻辑
7. **第三方 ERP**：联系商务，协调修改时间

## 第七步：灰度建议

- **是否建议灰度发布：** 是（如果可以）
- **推荐方案：** API 版本管理，而非直接变更
- **如果必须直接变更：**
  - 灰度范围：10% 用户
  - 灰度时长：24 小时
  - 监控指标：API 错误率、前端报错率
  - 回滚条件：错误率上升超过 1%

## 最终报告

### 变更概况
- 变更服务：订单服务
- 变更内容：GET /api/orders/:id 响应格式重构
- 变更类型：破坏性变更
- 影响调用方：5 个

### 推荐方案
**强烈建议使用 API 版本管理**，而非直接变更：
- 新增 v2 接口
- 保留 v1 接口 6 个月
- 调用方逐步迁移

### 关键行动
1. 与所有调用方协调发布时间
2. 优先处理第三方 ERP（协调难度最大）
3. 移动端需要提前发版

---

## 经验总结

1. **破坏性变更影响范围广**，应优先考虑向后兼容方案
2. **API 版本管理**是处理破坏性变更的最佳实践
3. **移动端发版延迟**是额外风险，需要提前规划
4. **第三方集成**是最难控制的环节，需要充分沟通
5. **TypeScript 类型**可以提前发现问题，是好事

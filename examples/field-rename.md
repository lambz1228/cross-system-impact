# 案例一：数据库字段重命名

## 场景描述

订单服务需要将 `orders` 表的 `total` 字段重命名为 `amount`，原因是与财务系统的字段命名保持一致。

**变更内容：**
- 字段：`orders.total` → `orders.amount`
- 类型：重命名
- 服务：订单服务
- 文件：`src/models/order.ts`, `migrations/20260608_rename_total.sql`

---

```
变更对象：
- 字段名称：orders.total -> orders.amount
- 变更类型：重命名

  - src/services/order.service.ts（业务逻辑）
```

## 第四步：影响评估

| # | 下游服务 | 影响等级 | 影响类型 | 失败模式 | 恢复难度 | 检测难度 |
|---|---------|---------|---------|---------|---------|---------|
| 1 | 支付服务 | Critical | 功能中断 | 消费消息时 total 为 null，扣款失败 | 简单 | 立即 |
| 2 | 计费服务 | High | 数据错误 | SQL 查询 total 列名错误 | 简单 | 立即 |

- Critical: 1 个服务

- **总计影响: 4 个服务**

## 第五步：风险分析

### 支付服务（Critical）

**依赖方式：** Kafka 消费 order.created，读取 total 字段计算扣款金额

**影响分析：**
- 字段重命名后，消息体中 total 字段消失
- 支付服务读取 total 为 null
- 金额校验失败，支付流程中断

**最坏情况：** 所有新订单无法支付，核心交易链路中断

**能否自动检测：** 能，支付服务有金额校验会立即报错

### 计费服务（High）

**依赖方式：** Python 服务直接查询 orders.total 列

**影响分析：**
- SQL 语句 `SELECT total FROM orders` 执行失败
- 佣金计算任务暂停
- 历史数据不受影响（迁移会处理）

**最坏情况：** 佣金数据延迟，不影响核心交易

**能否自动检测：** 能，SQL 执行失败会报错

### 报表服务（High）

**依赖方式：** Go 服务直连 MySQL，查询 orders.total

**影响分析：**
- 与计费服务类似，SQL 查询失败
- 日报生成任务失败

**最坏情况：** 报表数据延迟

### 前端（Medium）

**依赖方式：** HTTP API 返回 total 字段

**影响分析：**
- 订单详情页 total 字段显示 undefined
- 用户看到空白或 0

**最坏情况：** 显示异常，不影响核心功能

## 第七步：灰度建议

- **是否建议灰度发布：** 是
- **灰度范围：** 10% 订单流量
- **灰度时长：** 2 小时
- **灰度期间监控指标：**
  - 支付成功率
  - 订单创建成功率
  - 佣金计算延迟
  - 报表生成状态
- **回滚条件：**
  - 支付成功率下降超过 1%
  - 订单创建失败率上升
- **回滚方案：**
  1. 回滚订单服务数据库迁移
  2. 回滚订单服务代码
  3. 确认下游服务恢复正常

## 经验总结

1. **字段重命名是最常见的跨系统变更**，影响范围容易被低估
2. **Kafka 消费者数量**是关键信息，需要提前确认
3. **数据库直连的服务**容易被遗漏，需要主动排查
4. **灰度发布**可以有效控制风险

# 案例三：Kafka 消息 Schema 变更

## 场景描述

支付服务需要在 `payment.completed` 事件中新增 `currency` 字段，用于支持多币种支付。

**变更内容：**
- Topic：`payment.completed`
- 变更类型：新增字段
- 服务：支付服务
- 文件：`src/events/payment.event.ts`

**变更前：**
```json
{
  "order_id": "ord_123",
  "amount": 99.00,
  "paid_at": "2026-06-08T10:00:00Z",
  "payment_method": "alipay"
}
```

**变更后：**
```json
{
  "order_id": "ord_123",
  "amount": 99.00,
  "currency": "CNY",
  "paid_at": "2026-06-08T10:00:00Z",
  "payment_method": "alipay"
}
```

## 第五步：风险分析

### 结论：本次变更风险极低

**原因分析：**

1. **新增字段是向后兼容的**
   - 现有消费者会忽略不认识的字段
   - 不会导致报错或数据错误

2. **现有消费者不依赖新字段**
   - 订单服务：只用 order_id 和 amount
   - 计费服务：只用 amount
   - 通知服务：只用 order_id

3. **Kafka 消息格式特性**
   - JSON 格式天然支持向后兼容
   - 新增字段不会影响旧消费者

---

## 第六步：行动建议

### [低风险变更，无需紧急行动]

1. **可以正常发布**
   - 无需等待其他服务修改
   - 无需灰度发布

2. **建议的通知**
   - 通知相关团队：本次变更新增了 currency 字段
   - 说明：不影响现有功能，未来可选使用

3. **未来规划**
   - 如果未来要使用 currency 字段，需要：
     - 计费服务：修改佣金计算逻辑，支持多币种
     - 订单服务：存储 currency 字段

## 经验总结

1. **新增字段通常是安全的**
   - JSON 格式天然支持向后兼容
   - 消费者会忽略不认识的字段

2. **但要注意特殊情况**
   - 消费者是否有严格的 schema 校验？
   - 消费者是否使用了 strict mode？
   - 是否有 Schema Registry 强制校验？

3. **向后兼容 vs 向前兼容**
   - 新增字段：向后兼容（新生产者 → 旧消费者）
   - 删除字段：向后兼容（新生产者 → 旧消费者）
   - 重命名字段：不兼容
   - 修改字段类型：不兼容

---

## 对比：不同变更类型的风险

| 变更类型 | 风险等级 | 说明 |
|---------|---------|------|
| 新增字段 | 低 | 向后兼容 |
| 删除字段 | 高 | 旧消费者可能报错 |
| 重命名字段 | 高 | 不兼容 |
| 修改字段类型 | 高 | 不兼容 |
| 修改字段语义 | 中 | 静默数据错误 |

---

## Schema Registry 的作用

如果使用了 Confluent Schema Registry：

1. **兼容性检查**
   - BACKWARD_COMPATIBLE：新 schema 能读旧数据
   - FORWARD_COMPATIBLE：旧 schema 能读新数据
   - FULL_COMPATIBLE：双向兼容

2. **本次变更的兼容性**
   - 新增字段：BACKWARD_COMPATIBLE ✓
   - 需要在 Schema Registry 中注册新 schema

3. **建议**
   - 配置 Schema Registry 的兼容性模式
   - 在 CI/CD 中集成 schema 兼容性检查

# Cross-System Impact Analysis（跨系统变更影响分析）

## 用途

当一个变更可能影响多个服务时，系统化地识别哪些下游服务会受影响、影响程度如何、需要通知谁、怎么修。

本技能本身不做自动发现——它是**分析框架**。用户提供拓扑信息，技能按框架结构化输出影响报告。

## 核心问题

单仓库内的影响分析（LSP、grep、ref 工具）能回答：「这个函数被谁调了」。

本技能回答的是：「这个字段/接口/事件在公司哪些服务之间流动，改了会断几条链路」。

这两者的区别：

| | 单仓库分析 | 跨系统分析 |
| --- | --- | --- |
| 范围 | 当前仓库 | 多个仓库/服务 |
| 发现方式 | LSP、AST、grep | 用户提供拓扑 + 代码扫描 |
| 检测对象 | 函数调用、import | 字段名、API 契约、消息 schema、数据库列 |
| 能看到的 | 本仓库代码 | 跨服务依赖关系 |
| 看不到的 | 其他服务 | 其他服务（除非用户提供） |

## 何时使用

- 修改了 API 响应字段（JSON 结构变化）
- 修改了数据库字段名或类型
- 修改了 Kafka/RabbitMQ/消息队列的事件 schema
- 修改了共享库/SDK 的接口
- 修改了 Redis 缓存的 key 结构
- 修改了 Webhook payload 格式
- 任何可能影响下游消费者的变更
- 用户说「改这个字段会不会影响其他服务」「影响范围有多大」「需要通知谁」

## 使用示例

### 示例 1：字段重命名

用户输入：
> 我把订单服务的 `order.total` 改名为 `order.amount`，影响范围有多大？

分析流程：
1. 识别变更：字段重命名 `total` -> `amount`
2. 收集拓扑：查询哪些服务使用 `total` 字段
3. 追踪路径：沿消息队列、API、数据库读取链路追踪
4. 输出报告：按影响等级排序

### 示例 2：接口变更

用户输入：
> 我要给支付接口增加一个必填字段 `currency`，会影响谁？

分析流程：
1. 识别变更：请求字段新增必填项
2. 收集拓扑：查询哪些服务调用支付接口
3. 追踪路径：找到所有调用方
4. 输出报告：列出需要修改的调用方

## 框架流程

### 第一步：变更识别

从用户的 diff 或描述中提取：

```
变更对象：
- 字段/接口名称：order.total -> order.amount
- 变更类型：重命名 / 类型变更 / 删除 / 新增 / 语义变更
- 所属服务：订单服务
- 变更文件：src/models/order.ts
- 变更 diff 摘要：一行描述具体改动
```

**变更类型说明：**

| 类型 | 说明 | 典型影响 |
|------|------|---------|
| 重命名 | 字段名变更 | 下游读取失败 |
| 类型变更 | 如 string -> number | 类型不匹配报错 |
| 删除 | 字段移除 | 下游读取 null/undefined |
| 新增必填 | 新增必填字段 | 调用方缺少参数 |
| 语义变更 | 含义变化 | 静默数据错误 |

### 第二步：拓扑收集

**这是关键步骤。** 用户需要提供（或确认）服务间的数据流拓扑。

信息来源（按优先级）：
1. 用户手动描述
2. 消息队列配置文件（Kafka topic 定义、RabbitMQ exchange）
3. API 定义文件（OpenAPI、protobuf、GraphQL schema）
4. 数据库 schema（表结构、外键）
5. 配置文件中的服务地址和路由
6. 代码中的 HTTP 调用、消息生产/消费代码

拓扑描述格式见 `references/topology-format.md`。

**拓扑信息完整性检查：**

- [ ] 是否列出了所有直接依赖的服务？
- [ ] 是否包含了消息队列的 topic/exchange？
- [ ] 是否包含了 API 接口路径？
- [ ] 是否包含了数据库表的读写关系？
- [ ] 是否包含了缓存的使用情况？

### 第三步：传播路径追踪

沿着拓扑图，从变更点出发，追踪所有传播路径：

```
order.total（订单服务）
  |
  +--[Kafka: order.created]--> 支付服务（消费 total 字段用于扣款）
  |                              |
  |                              +--[HTTP]--> 第三方支付网关
  |
  +--[Kafka: order.created]--> 计费服务（SELECT total 列算佣金）
  |
  +--[Kafka: order.created]--> 财务报表服务（消费 total 生成日报）
  |
  +--[HTTP API]--> 前端（显示订单总额）
  |
  +--[Redis 缓存]--> 订单详情缓存（total 作为返回字段）
  |
  x  物流服务（不使用 total 字段，无影响）
```

**追踪规则：**

1. **直接依赖**：消息体包含该字段、API 返回该字段、SQL 查询该列
2. **间接依赖**：通过中间服务传递（A -> B -> C，改 A 影响 C）
3. **隐式依赖**：定时任务、报表系统、数据同步管道

### 第四步：影响评估

对每个受影响的下游服务评估：

| 评估维度 | 说明 |
| --- | --- |
| **影响等级** | Critical / High / Medium / Low / None |
| **影响类型** | 功能中断 / 数据错误 / 性能下降 / 无影响 |
| **失败模式** | 字段缺失返回 null / 类型不匹配报错 / 静默数据错误 |
| **恢复难度** | 简单（改一个字段名）/ 中等（需要迁移）/ 复杂（需要多服务协调） |
| **检测难度** | 立即发现（报错）/ 延迟发现（数据错误但不报错）/ 难发现（静默错误） |

**影响等级判定规则：**

- **Critical**：核心链路中断（支付、订单、认证、登录）
- **High**：重要功能异常（计费、库存、通知、搜索）
- **Medium**：非核心功能受损（报表、推荐、日志、统计）
- **Low**：边缘功能（非关键展示、缓存延迟、辅助功能）
- **None**：无影响

**检测难度说明：**

| 难度 | 特征 | 示例 |
|------|------|------|
| 立即发现 | 有类型校验、必填校验、单元测试 | TypeScript 类型错误、SQL 报错 |
| 延迟发现 | 运行时报错但不崩溃 | API 返回 null、日志中有警告 |
| 难发现 | 静默数据错误 | 计算结果错误、显示内容错误 |

### 第五步：输出报告

格式见 `references/report-template.md`。

### 第六步：行动建议

按影响等级从高到低，给出：

1. **需要立即通知的团队**
   - 联系方式（群组、IM、邮件）
   - 需要同步修改的内容
   - 截止时间

2. **需要同步修改的服务**
   - 具体修改内容
   - 修改优先级
   - 验证方式

3. **需要观察的指标**
   - 关键业务指标（支付成功率、订单量等）
   - 技术指标（错误率、延迟、CPU/内存）
   - 告警配置

4. **灰度发布建议**
   - 是否需要灰度
   - 灰度范围
   - 灰度时长
   - 回滚条件

5. **回滚方案**
   - 回滚步骤
   - 回滚影响
   - 数据修复方案

## 与其他 Skill 的关系

- **security-audit**：跨系统安全审计（认证链路、权限传递）
- **database-design**：数据库 schema 变更的跨服务影响
- **migration-planner**：多服务协调迁移计划
- **api-design**：API 契约变更的兼容性分析

## 最佳实践

1. **提前规划**：大范围变更前先做影响分析
2. **完整拓扑**：拓扑信息越完整，分析越准确
3. **分批变更**：影响范围大时，分批次、灰度发布
4. **充分沟通**：通知所有受影响的团队
5. **监控验证**：变更后密切关注关键指标
6. **文档更新**：变更完成后更新架构文档

## 局限性

本技能是分析框架，不能自动发现：

- 用户未描述的服务
- 未文档化的消息队列消费关系
- 隐式的字段依赖（如下游服务用字符串拼接字段名）
- 运行时动态路由的服务
- 第三方 SaaS 服务的内部实现

**越完整的拓扑输入，输出越准确。**

# 常见数据流模式

识别变更影响时，按以下模式检查每种可能的传播路径。

## 1. 消息队列（异步）

### Kafka

**检查要点：**
- 检查 Topic：哪些 Producer 生产，哪些 Consumer 消费
- 字段级影响：消息体里的字段名变更 = 所有消费者都受影响
- 延迟发现：消费者可能静默丢弃未知字段，导致数据丢失不报错

**常见场景：**

| 场景 | 字段变更影响 | 检测难度 |
|------|-------------|---------|
| 事件驱动架构 | 所有消费者都受影响 | 中等 |
| CQRS 模型 | 读模型可能数据不一致 | 难 |
| 事件溯源 | 历史事件重放失败 | 难 |

**检查清单：**
- [ ] 这个字段在消息体里吗？
- [ ] 有多少个消费者组？
- [ ] 消费者有没有对未知字段的容错逻辑？
- [ ] 消费者失败后有重试机制吗？重试会不会放大问题？
- [ ] 消费者是否有 dead letter queue 处理失败消息？
- [ ] 消息格式是否有 schema 注册中心（如 Confluent Schema Registry）？

### RabbitMQ / RocketMQ

**检查要点：**
- 检查 Exchange 和 Queue 的绑定关系
- 路由 Key 变更可能影响多个 Queue
- 消息体 schema 变更影响所有消费者

**常见场景：**

| 场景 | 字段变更影响 | 检测难度 |
|------|-------------|---------|
| 发布/订阅模式 | 所有订阅者都受影响 | 中等 |
| 路由模式 | 特定路由的消费者受影响 | 中等 |
| RPC 模式 | 调用方和被调用方都受影响 | 立即 |

**检查清单：**
- [ ] Exchange 类型是什么（direct/topic/fanout）？
- [ ] 路由 Key 是否包含被修改的字段？
- [ ] 消息是否有 TTL 设置？过期后如何处理？
- [ ] 是否有优先级队列？字段变更是否影响优先级判断？

## 2. HTTP API（同步）

### REST API

**检查要点：**
- 修改响应字段 = 所有调用方都受影响
- 修改请求字段 = 所有调用方都需要改
- 修改状态码 / 错误格式 = 客户端异常处理可能失效

**常见场景：**

| 场景 | 字段变更影响 | 检测难度 |
|------|-------------|---------|
| 公开 API | 外部调用方都受影响 | 难（无法控制） |
| 内部 API | 内部服务受影响 | 中等 |
| BFF 模式 | 前端受影响 | 立即 |

**检查清单：**
- [ ] 有多少个调用方？
- [ ] 调用方是内部服务还是外部客户？
- [ ] 有没有 API 版本管理（v1/v2）？
- [ ] 调用方有没有契约测试（contract test）？
- [ ] 是否有 API 网关做字段转换？
- [ ] 是否有 OpenAPI/Swagger 文档需要更新？

### GraphQL

**检查要点：**
- 修改 field name = 查询该 field 的所有客户端都报错
- 删除 field = 客户端报 null 或 missing field
- 修改 type = 类型不匹配报错

**常见场景：**

| 场景 | 字段变更影响 | 检测难度 |
|------|-------------|---------|
| 公开 API | 外部客户端都受影响 | 难 |
| 内部 BFF | 前端受影响 | 立即 |
| Federation | 子图变更影响整体 | 中等 |

**检查清单：**
- [ ] 哪些查询使用了这个 field？
- [ ] 是否有 @deprecated 指令？
- [ ] 是否有 persisted queries（持久化查询）？
- [ ] 客户端是否有 codegen 生成的类型？

### gRPC / protobuf

**检查要点：**
- 字段编号变更 = 协议不兼容
- 重命名字段（不影响编号）= 客户端侧兼容
- 删除字段 = 客户端可能读到默认值

**常见场景：**

| 场景 | 字段变更影响 | 检测难度 |
|------|-------------|---------|
| 微服务间通信 | 下游服务受影响 | 中等 |
| 客户端 SDK | 客户端受影响 | 难 |
| 流式 RPC | 双向都受影响 | 中等 |

**检查清单：**
- [ ] 字段编号是否变更？
- [ ] 是否有 reserved 字段？
- [ ] 客户端是否使用 old compiler 版本？
- [ ] 是否有 backward/forward compatibility 测试？

## 3. 数据库（共享）

### 共享数据库

**检查要点：**
- 多个服务连同一个数据库：字段名变更影响所有直连的服务
- 检查哪些服务有数据库连接权限

**常见场景：**

| 场景 | 字段变更影响 | 检测难度 |
|------|-------------|---------|
| 单库多服务 | 所有直连服务受影响 | 中等 |
| 读写分离 | 读副本受影响 | 中等 |
| 分库分片 | 特定分片受影响 | 难 |

**检查清单：**
- [ ] 哪些服务直接连这个数据库？
- [ ] 是否有数据库连接池配置？
- [ ] 是否有 ORM 模型定义需要更新？
- [ ] 是否有数据库迁移脚本需要同步？

### 数据同步

**检查要点：**
- CDC（Change Data Capture）：字段变更通过 binlog/WAL 同步到下游
- ETL 管道：定时任务把数据搬到数据仓库
- 数据复制：读写分离、主从同步

**常见场景：**

| 场景 | 字段变更影响 | 检测难度 |
|------|-------------|---------|
| CDC 到 ES | 搜索索引受影响 | 中等 |
| ETL 到数仓 | 报表数据受影响 | 难 |
| 实时同步到 Redis | 缓存数据受影响 | 中等 |

**检查清单：**
- [ ] 有没有 CDC 管道（Debezium、Canal）？
- [ ] 有没有 ETL 任务定时读这个表？
- [ ] 有没有数据同步到 ES/MongoDB/ClickHouse？
- [ ] 有没有报表系统直接查这个表？
- [ ] CDC 管道是否有字段过滤配置？

## 4. 缓存

### Redis

**检查要点：**
- 缓存 key 包含字段名：key 过期前旧格式和新格式共存
- 缓存 value 包含字段：消费方读到旧缓存可能忽略新字段
- 缓存失效策略影响问题持续时间

**常见场景：**

| 场景 | 字段变更影响 | 检测难度 |
|------|-------------|---------|
| Session 缓存 | 用户状态异常 | 中等 |
| 业务数据缓存 | 数据不一致 | 中等 |
| 分布式锁 | 锁失效 | 难 |

**检查清单：**
- [ ] 有没有缓存这个字段？
- [ ] 缓存过期时间多久？
- [ ] 缓存 miss 时有 fallback 到数据库吗？
- [ ] 是否需要主动失效缓存？
- [ ] 缓存 key 格式是否包含字段名？

## 5. 共享库 / SDK

### 内部 npm/pip/maven 包

**检查要点：**
- 版本升级后所有依赖方都需要更新
- Breaking change 需要 major version bump
- 有些服务可能不主动升级

**常见场景：**

| 场景 | 字段变更影响 | 检测难度 |
|------|-------------|---------|
| 公共组件库 | 所有使用方受影响 | 中等 |
| API SDK | 所有调用方受影响 | 中等 |
| 工具库 | 使用方可能受影响 | 难 |

**检查清单：**
- [ ] 这个变更在共享库里吗？
- [ ] 有多少个服务依赖这个包？
- [ ] 这些服务的升级频率如何？
- [ ] 有没有 lockfile 锁定版本？
- [ ] 是否有 CHANGELOG 需要更新？
- [ ] 是否需要发布新版本？

## 6. 定时任务 / 批处理

**检查要点：**
- Cron job、Airflow DAG、定时脚本可能使用被变更的字段
- 这类依赖最隐蔽，因为不在实时链路里

**常见场景：**

| 场景 | 字段变更影响 | 检测难度 |
|------|-------------|---------|
| 日报/周报 | 统计数据错误 | 难 |
| 数据清理 | 清理逻辑错误 | 难 |
| 数据同步 | 同步数据错误 | 难 |

**检查清单：**
- [ ] 有没有定时任务读这个字段？
- [ ] 有没有报表/BI 系统查询这个字段？
- [ ] 有没有数据导出任务使用这个字段？
- [ ] 定时任务的执行频率是什么？
- [ ] 任务失败后有告警吗？

## 7. 前端 / 客户端

**检查要点：**
- API 返回字段变更直接影响前端显示
- TypeScript 类型不匹配会在编译时报错（好情况）
- JavaScript 动态取值不会报错但显示 undefined（坏情况）
- 移动端 App 有发布延迟，旧版本可能长期存在

**常见场景：**

| 场景 | 字段变更影响 | 检测难度 |
|------|-------------|---------|
| Web 应用 | 显示异常 | 立即 |
| 移动 App | 显示异常（旧版本） | 延迟 |
| 小程序 | 显示异常 | 延迟 |
| 第三方集成 | 集成失败 | 难 |

**检查清单：**
- [ ] 有前端在用这个字段吗？
- [ ] 前端是 TypeScript 还是 JavaScript？
- [ ] 移动端有这个功能吗？最新版本发布了吗？
- [ ] 有小程序吗？
- [ ] 是否有字段映射/转换层？
- [ ] 是否有 feature flag 控制新旧字段？

## 8. 第三方服务

**检查要点：**
- Webhook 接收方可能依赖特定字段格式
- 外部 API 调用方可能依赖特定字段
- SaaS 集成可能有字段要求

**常见场景：**

| 场景 | 字段变更影响 | 检测难度 |
|------|-------------|---------|
| Webhook 推送 | 第三方处理失败 | 难 |
| 外部 API 调用 | 调用失败 | 立即 |
| SaaS 集成 | 集成异常 | 中等 |

**检查清单：**
- [ ] 有没有 Webhook 推送这个字段？
- [ ] 有没有外部系统调用这个 API？
- [ ] 有没有 SaaS 集成使用这个字段？
- [ ] 第三方是否有字段验证？
- [ ] 是否需要通知第三方？

## 影响传播模式总结

| 传播模式 | 特点 | 检测难度 | 典型场景 |
|---------|------|---------|---------|
| 同步调用 | 立即失败 | 立即 | HTTP API、gRPC |
| 异步消息 | 延迟失败 | 中等 | Kafka、RabbitMQ |
| 数据库直连 | SQL 报错 | 立即 | 共享数据库 |
| 数据同步 | 数据不一致 | 难 | CDC、ETL |
| 缓存 | 数据不一致 | 中等 | Redis |
| 定时任务 | 静默错误 | 难 | Cron、Airflow |
| 前端 | 显示异常 | 立即/延迟 | Web、App |

## 优先级建议

根据影响传播模式，建议按以下优先级检查：

1. **同步调用**：立即检查，影响最直接
2. **异步消息**：重点检查，消费者数量可能很多
3. **数据库直连**：检查所有直连服务
4. **数据同步**：检查 CDC/ETL 管道
5. **缓存**：检查缓存失效策略
6. **定时任务**：最容易遗漏，需要主动排查
7. **前端**：检查所有客户端
8. **第三方**：最难控制，需要提前沟通

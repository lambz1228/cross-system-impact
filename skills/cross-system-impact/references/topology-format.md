# 拓扑描述格式

用户以结构化方式描述服务间的数据流关系。格式越完整，分析越准确。

## 基础格式

```md
## 服务拓扑

### 服务列表
| 服务名 | 技术栈 | 仓库位置 | 负责团队 | 优先级 |
| --- | --- | --- | --- | --- |
| 订单服务 | Node.js | git@xxx/order-service | 交易组 | P0 |
| 支付服务 | Java | git@xxx/payment-service | 支付组 | P0 |
| 计费服务 | Python | git@xxx/billing-service | 财务组 | P1 |
| 报表服务 | Go | git@xxx/report-service | 数据组 | P2 |

**字段说明：**
- 服务名：唯一标识，后续引用使用
- 技术栈：Node.js / Java / Python / Go / 其他
- 仓库位置：Git 仓库地址（便于定位代码）
- 负责团队：负责该服务的团队
- 优先级：P0（核心）/ P1（重要）/ P2（一般）/ P3（边缘）

### 数据流

#### 订单服务 -> 支付服务
- 方式：Kafka
- Topic：order.created
- 关键字段：order_id, total, user_id, payment_method
- 触发条件：订单创建成功后
- 延迟要求：< 1s
- 失败处理：重试 3 次，失败后告警

#### 订单服务 -> 前端
- 方式：HTTP API
- 接口：GET /api/orders/:id
- 返回字段：id, total, status, items, created_at
- 调用方：前端 Web、移动端 App
- 认证方式：JWT Token
- 限流：100 QPS

#### 支付服务 -> 计费服务
- 方式：Kafka
- Topic：payment.completed
- 关键字段：order_id, amount, currency, paid_at
- 触发条件：支付成功后
- 延迟要求：< 5s
- 失败处理：死信队列 + 人工介入

### 共享数据
| 字段 | 写入方 | 读取方 | 存储位置 | 一致性要求 |
| --- | --- | --- | --- | --- |
| order.total | 订单服务 | 支付服务、计费服务、财务报表 | MySQL orders.total | 强一致 |
| user.id | 用户服务 | 所有服务 | MySQL + Redis | 最终一致 |
| product.price | 商品服务 | 订单服务、库存服务 | MySQL + Redis Cache | 最终一致 |

### 事件定义
| 事件名 | 生产者 | 消费者 | Schema 位置 |
| --- | --- | --- | --- |
| order.created | 订单服务 | 支付服务、计费服务 | schema/orders.json |
| payment.completed | 支付服务 | 计费服务、通知服务 | schema/payments.json |
| user.registered | 用户服务 | 营销服务、CRM | schema/users.json |
```

## 最小格式（快速分析）

如果用户只有粗略信息，可以用简化格式：

```md
## 简要拓扑

订单服务通过 Kafka 发 order.created 事件，支付服务和计费服务消费。
订单总额字段 total 在支付服务里用于扣款，在计费服务里算佣金。
前端通过 HTTP API 读取订单数据展示给用户。
```

即使信息不完整也能开始分析，但会明确标注哪些部分需要补充。

## 完整格式（详细分析）

对于关键系统或大范围变更，建议使用完整格式：

```md
## 详细拓扑

### 架构概览
[前端] --> [API 网关] --> [订单服务] --> [支付服务]
                              |              |
                              v              v
                         [计费服务]    [第三方支付]

### 服务详情

#### 订单服务
- 类型：核心业务服务
- 技术栈：Node.js + Express
- 数据库：MySQL (orders_db)
- 缓存：Redis (order_cache)
- 消息队列：Kafka Producer
- 依赖服务：用户服务、商品服务
- 被依赖方：支付服务、计费服务、报表服务、前端

#### 支付服务
- 类型：核心业务服务
- 技术栈：Java + Spring Boot
- 数据库：MySQL (payments_db)
- 消息队列：Kafka Consumer + Producer
- 外部依赖：支付宝、微信支付、银联
- 被依赖方：计费服务、通知服务

### 数据流详情

#### 1. 下单流程（同步 + 异步）
前端 -> API 网关 -> 订单服务（创建订单）
订单服务 -> Kafka（order.created）
支付服务 <- Kafka（消费 order.created）
支付服务 -> 第三方支付（发起扣款）
支付服务 -> Kafka（payment.completed）
计费服务 <- Kafka（消费 payment.completed）

#### 2. 查询流程（同步）
前端 -> API 网关 -> 订单服务（查询订单）
订单服务 -> MySQL（读取订单）
订单服务 -> Redis（读取缓存）

### 数据依赖矩阵
| 字段 | 生产者 | 消费者1 | 消费者2 | 消费者3 |
|------|--------|---------|---------|---------|
| order.id | 订单服务 | 支付服务 | 计费服务 | 报表服务 |
| order.total | 订单服务 | 支付服务 | 计费服务 | - |
| payment.status | 支付服务 | 订单服务 | 通知服务 | 报表服务 |
```

## 信息来源优先级

当用户不确定拓扑时，建议按以下顺序查找：

1. **消息队列控制台**：Kafka topics、RabbitMQ exchanges 的生产者和消费者列表
2. **API 网关配置**：路由规则、上游/下游映射
3. **数据库**：表的读写来源（谁写、谁读）
4. **代码仓库**：HTTP 调用、消息生产/消费代码
5. **架构文档**：如果有维护的话
6. **口头询问**：问相关服务的负责人

## 拓扑信息质量评估

| 质量等级 | 特征 | 分析准确性 |
|---------|------|-----------|
| 优秀 | 完整的服务列表 + 数据流 + 字段级依赖 | 90%+ |
| 良好 | 服务列表 + 主要数据流 | 70-90% |
| 一般 | 只有粗略的服务关系 | 50-70% |
| 较差 | 只知道部分服务名 | < 50% |

## 自动化发现的可能性（研究方向）

未来可以通过以下方式自动收集拓扑信息：

- 解析 Kafka/RabbitMQ 配置文件和控制台 API
- 扫描 OpenAPI/protobuf 定义文件
- 分析代码中的 HTTP 客户端调用（fetch、axios、requests）
- 分析 ORM 模型和迁移文件中的字段
- 扫描数据库连接配置
- 使用 LSP 跨仓库索引

这些目前没有现成工具串起来，但每个环节都有可行的技术路径。

## 常见问题

### Q: 不确定某个服务是否依赖被修改的字段怎么办？

A: 在拓扑中标注「不确定」，分析时会将其列为「待确认」，并在报告的「信息缺口」部分提醒确认。

### Q: 服务太多，没法一次性描述完怎么办？

A: 可以分阶段描述：先描述与变更直接相关的核心服务，再逐步补充间接依赖的服务。

### Q: 有些依赖关系是动态的（如配置中心控制路由）怎么办？

A: 在拓扑中说明动态路由的规则，分析时会考虑多种可能的路由情况。

### Q: 第三方 SaaS 服务怎么描述？

A: 将其作为一个服务节点，标注为「外部服务」，说明数据流向即可。不需要了解其内部实现。
